# Week 3 실습 기록

목표
플레이어 캐릭터를 키보드로 움직이기 구현

AI 대화 기록

Q1: Pygame 코드 각 줄이 무엇을 하는지 설명해줘

AI 답변: 코드 전체를 준비 → 게임 루프 → 이벤트 감지 → 그리기 → 속도 조절 → 종료의 6단계 구조로 설명
주요 내용:

pygame.init() : 엔진 시동
while running : 게임 루프 (모든 게임의 핵심 구조)
pygame.event.get() : 키 입력·마우스 등 이벤트 감지
screen.fill(WHITE) : 매 프레임 화면을 지워 잔상 방지
clock.tick(60) : 초당 60프레임으로 속도 제한


Q2: screen.fill(WHITE)를 삭제하면 어떻게 달라지는가?

AI 답변: 원이 고정된 현재 코드에서는 배경만 검게 바뀌고, 원이 움직이는 코드에서는 이전 프레임이 지워지지 않아 잔상이 쌓임
핵심 원리: pygame은 "지우기" 기능이 없어서 screen.fill()로 전체를 덮어씌우는 것이 곧 지우는 행위

애니메이션의 "지우고 → 다시 그리고 → 보여주기" 3단계 원리 이해


Q3: clock.tick(60)을 clock.tick(5)로 바꾸면 어떻게 달라지는가?

AI 답변: 초당 60프레임 → 초당 5프레임으로 줄어들어 화면이 뚝뚝 끊기고, X 버튼 같은 입력 반응도 최대 200ms까지 느려짐
프레임별 간격: tick(60) = 16ms, tick(5) = 200ms

FPS 숫자가 게임 부드러움과 입력 반응 속도 모두에 영향을 준다는 것 이해


Q4: 방향키 입력을 추가해서 원이 5픽셀씩 움직이게 하려면?

AI 답변: 세 가지를 변경

원 위치를 변수 x, y = 400, 300으로 분리
pygame.key.get_pressed()로 방향키 감지 후 x, y 값 변경
pygame.draw.circle()의 고정 좌표를 변수 (x, y)로 교체

event.get() vs key.get_pressed() 차이: 전자는 눌렀다 뗀 순간만, 후자는 누르는 동안 계속 감지
적용 결과: 방향키로 원 이동 구현 성공


Q5: y -= 5가 왜 위쪽으로 이동하는가?

AI 답변: 컴퓨터 화면은 수학 좌표계와 반대로, 왼쪽 위 모서리가 (0, 0)이고 아래로 갈수록 y가 커지는 구조
유래: 오래된 CRT 모니터가 전자빔을 왼쪽 위에서 시작해 아래로 주사하던 방식에서 고정됨
pygame뿐 아니라 CSS, HTML Canvas 등 거의 모든 화면 관련 기술이 동일한 방식 사용




시행착오 및 해결 과정

시행착오 1: 들여쓰기 오류

PPT에 있는 코드를 그대로 복사해 붙여넣었더니 들여쓰기가 없어 IndentationError 오류 발생함
원인: while 블록 안의 for, if, 그리기 코드에 들여쓰기 누락
해결: while 내부 코드를 4칸씩 들여쓰기 적용. 중첩 블록은 4칸씩 추가

시행착오 2: 화면 좌표계 혼동

y -= 5가 위로 이동하는 게 직관적으로 이해가 안 됨
원인: 수학에서 배운 y축 방향(위로 증가)과 화면 좌표계(아래로 증가)가 반대
해결: 화면의 원점(0,0)이 왼쪽 위에 있다는 것을 이해

시행착오 3: 경계 처리 시 원이 잘리는 문제

처음에 x = max(0, min(800, x))로 작성했더니 원이 벽에 닿을 때 절반이 잘림
원인: 좌표가 원의 중심이라 x=0이면 왼쪽 절반이 화면 밖으로 나감
해결: 반지름(50)을 고려해 max(RADIUS, min(SCREEN_W - RADIUS, x))로 수정



배운 점

이번 주 코드로 인해 게임 프레임과 FPS에 대해 정확하게 이해하게 되었다. clock.tick() 숫자가 낮을수록 FPS가 낮아져 CPU는 사용은 아끼지만 
화면이 끊기고 입력 반응도 느려진다. 반면 FPS가 높으면 화면도 부드럽게 움직이고 입력 반응도 빠르지만 CPU 사용량이 높아진다는 걸 
AI와 문답을 통해 배웠다.

또 프로그래밍에서의 좌표계는 우리가 학교에서 배우던 수학 좌표계와는 달라서 왼쪽 위가 (0,0)이고 아래로 갈수록 y가 증가한다는 것도 새롭게 알게 되었다. 그래서 원을 움직이게 하려면 - 좌표를 입력해야 했다.
